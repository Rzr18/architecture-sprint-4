# Мотивация

* Добавление мониторинга в систему является критически важным шагом для обеспечения стабильности и производительности системы. С учетом того, что оформление заказов через API стало основным каналом взаимодействия с клиентами, необходимо иметь полное представление о состоянии системы. 
    
* Мониторинг позволит:
    + Выявлять узкие места: С помощью мониторинга можно быстро обнаруживать проблемы с производительностью, такие как задержки в обработке заказов или сбои в API.
	+ Оперативно реагировать на критические проблемы.
    + Улучшить пользовательский опыт: Понимание того, как пользователи взаимодействуют с системой, поможет оптимизировать процессы и повысить удовлетворенность клиентов.
    + Поддерживать репутацию компании: Своевременное выявление и устранение проблем поможет избежать негативных отзывов и сохранить контракты с партнерами.
    + Обосновать модернизацию: Наличие данных о производительности системы поможет обосновать необходимость выделения ресурсов на улучшение инфраструктуры и процессов.

# Выбор подхода к мониторингу

У нас система, где важно как измерять работу API-сервисов (пользовательские запросы, задержки), так и инфраструктуру (RabbitMQ, PostgreSQL). Поэтому:

1. **Бизнес-метрики**
		
1. **Для API-сервисов (shopApi, CRM API, MES API)**:
    + Логично придерживаться **RED**.
    + Важны количество запросов, задержки и ошибки.

2. **Для инфраструктуры (RabbitMQ, базы данных, S3-хранилище)**:
    + Подход **USE**: Utilization (например, CPU, память), Saturation (очереди запросов, backlog сообщений) и Errors (сбоев в соединении, отказы, dead-letter).
    + Для RabbitMQ отдельно важно считать сообщения в очередях, пропускную способность.

Таким образом, мы комбинируем подходы **RED** для сервисов и **USE** для баз данных и очередей.

# Метрики и их назначение

Ниже выбран перечень ключевых метрик, сгруппированных по компонентам системы. Для каждой метрики укажем:

* Бизнес-логика

1. **Время нахождения заказа в каждом из статусов**
    + Позволит определить этапы, наиболее подверженные задержкам.
    + *Ярлыки*: `state_name`.

* RabbitMQ

1. **Number of dead-letter-exchange letters**
    + Показатель количества сообщений, «забракованных» системой (ошибки при обработке, неправильный формат). Если число растёт — проблема в бизнес-логике или формате сообщений.
    + *Ярлыки*: `exchange_name`, `queue_name`, `environment` (dev/release/prod).

2. **Number of messages in flight**
    + Показывает, сколько сообщений одновременно обрабатывается/ждёт в очереди. При аномальном росте — сервис-потребитель не успевает обрабатывать заказы.
    + *Ярлыки*: `queue_name`, `environment`.

3. **Message delivery rate** (сообщений в секунду)
    + Чтобы понимать пропускную способность и общий объём событий (особенно B2B-вход).
    + *Ярлыки*: `exchange_name` (если несколько exchange для B2B/B2C), `environment`.

* Internet Shop (shopApi)

1. **Number of requests (RPS) for internet shop API**
    + Основной показатель нагрузки от B2C-пользователей.
    + *Ярлыки*: `endpoint` (если нужно детально понимать, какие методы вызываются чаще), `environment`.

2. **Response time (latency) for shop API**
    + Ключевой показатель UX для покупателей. Задержка увеличивается — падает конверсия в заказы.
    + *Ярлыки*: `endpoint`, `http_status`, `environment`.

3. **Number of HTTP 500 for shop API**
    + Критические ошибки для пользовательских операций. Рост — риск ухудшения репутации и оттока клиентов.
    + *Ярлыки*: `endpoint`, `environment`.

4. **CPU % for shop API** / **Memory Utilisation for shop API**
    + Важно понимать, когда под нагрузкой сервис начинает терять производительность. При регулярном превышении 80–90% стоит задуматься о масштабировании.
    + *Ярлыки*: `host`, `environment`.

5. **Number of simultanious sessions**
    + Видеть, сколько одновременно активных пользователей B2C. Использовать для capacity planning.
    + *Ярлыки*: `environment`.

* CRM API

1. **Number of requests (RPS) for CRM API**
    + Менеджеры могут массово обрабатывать заказы, плюс есть потенциал автоматических интеграций. Мониторим нагрузку.
    + *Ярлыки*: `environment`, `method` (если нужно различать GET/POST и т.д.).

2. **Response time (latency) for CRM API**
    + Чем быстрее CRM отвечает, тем эффективнее работа менеджеров. Высокая задержка может приводить к недовольству внутреннего пользователя.
    + *Ярлыки*: `endpoint`, `environment`.

3. **Number of HTTP 500 for CRM API**
    + Возможные проблемы в логике управления заказами, критично для B2B-партнёров, которые создают заказы через этот путь.
    + *Ярлыки*: `endpoint`, `environment`.

4. **CPU % и Memory Utilisation**
    + Следить за ресурсопотреблением.
    + *Ярлыки*: `host`, `environment`.

5. **Number of simultanious sessions** (CRM)
    + Видеть пиковую активность продавцов, менеджеров. Может влиять на планирование нагрузки, особенно в дневные часы.
    + *Ярлыки*: `environment`.

* MES API

1. **Number of requests (RPS) for MES API**
    + Понимать, с какой частотой операторы совершают операции (старт производства, апдейты и т.д.).
    + *Ярлыки*: `environment`, `method`, `endpoint`.

2. **Response time (latency) for MES API**
    + Критично, так как операторы не должны тратить время на ожидание загрузки дашборда, списков заказов.
    + *Ярлыки*: `endpoint`, `environment`.

3. **Number of HTTP 500 for MES API**
    + Сигнал о сбоях, которые мешают выполнению заказов.
    + *Ярлыки*: `endpoint`, `environment`.

4. **CPU % и Memory Utilisation** (MES API)
    + Отслеживать ресурсную загруженность MES, которая часто становится узким местом.
    + *Ярлыки*: `host`, `environment`.

5. **Number of simultanious sessions** (MES)
    + Количество операторов, активно работающих в системе, помогает планировать пропускную способность фронтенда/бэкенда.
    + *Ярлыки*: `environment`.

* Базы данных (Shop DB, MES DB)

1. **Memory Utilisation** (для Shop DB и MES DB)
    + Чтобы понимать, достаточно ли оперативной памяти, нет ли свопинга и падения производительности.
    + *Ярлыки*: `db_instance`, `environment`.

2. **Number of connections** (Shop DB, MES DB)
    + При большом числе соединений есть риск исчерпания connection pool. Если растёт, нужно масштабироваться или оптимизировать запросы.
    + *Ярлыки*: `db_instance`, `environment`.

3. **Size of shop db instance / Size of MES db instance**
    + Прогнозировать, когда понадобится увеличивать диск (особенно при хранении большого числа заказов/истории).
    + *Ярлыки*: `environment`.

4. **DB query latency** (по возможности)
    + Важно отслеживать, сколько реально занимают запросы в БД. Рост времени запроса негативно сказывается на API.
    + *Ярлыки*: `db_instance`, `query_type` (чтение/запись).

* Хранилище 3D-файлов (S3-based)

1. **Size of S3 storage**
    + Рост объёма хранилища может влиять на стоимость, нужно понимать тренд использования.
    + *Ярлыки*: `environment`.

2. **Number of upload errors**
    + Если по каким-то причинам 3D-файлы не сохраняются, это напрямую влияет на заказы.
    + *Ярлыки*: `environment`, `bucket_name` (если ведётся разделение).

# План действий

Для успешной реализации мониторинга системы необходимо выполнить следующие высокоуровневые задачи:

* Определение требований к мониторингу:
   + Провести встречу с ключевыми заинтересованными сторонами для сбора требований к метрикам и отчетам.
   + Согласовать список метрик, которые будут отслеживаться, и их важность для бизнеса.

* Выбор и настройка инструментов мониторинга:
   + Исследовать и выбрать подходящие инструменты для мониторинга.
   + Настроить интеграцию выбранных инструментов с существующими системами и API.

* Создание инстанса time-series базы данных:
   + Создать инстанс базы данных для хранения временных рядов.
   + Настроить автоматическое сохранение метрик в базе данных.

* Разработка и внедрение агентов мониторинга:
   + Разработать или настроить существующие агенты для сбора метрик с различных компонентов системы.
   + Обеспечить корректную передачу данных в систему мониторинга.

* Настройка алертов и уведомлений:
   + Определить пороговые значения для ключевых метрик и настроить алерты для уведомления команды о потенциальных проблемах.
   + Настроить интеграцию с системами уведомлений для оперативного реагирования.

* Тестирование и валидация системы мониторинга:
   + Провести тестирование системы мониторинга, чтобы убедиться в корректности сбора и отображения метрик.
   + Валидация работы алертов и уведомлений.

* Обучение команды:
   + Провести обучение для команды по использованию инструментов мониторинга и интерпретации данных.
   + Обсудить процесс реагирования на алерты и управление инцидентами.
